**Мета роботи:**

Реалізувати в системі управління задачами автентифікацію, авторизацію та рольову модель доступу (RBAC), використовуючи сервіс Supabase Auth та політики Row Level Security (RLS) у PostgreSQL.

**1. Модель користувача і безпечне зберігання паролів**

Завдяки використанню Supabase (BaaS) реалізовано промисловий стандарт безпеки:

- **Схема auth.users:** Облікові дані (email, хеш пароля) зберігаються у захищеній системній схемі, до якої немає прямого доступу з фронтенду.
- **Криптографія:** Паролі автоматично хешуються на стороні сервера Supabase з використанням стійких алгоритмів (Bcrypt) і ніколи не передаються чи не зберігаються у відкритому вигляді.
- **Таблиця public.profiles:** Додаткові дані (ім'я, роль user/admin, статус is\_blocked) зберігаються у публічній схемі та пов'язані з auth.users через id (UUID).
- **Автоматизація:** Після успішної реєстрації через supabase.auth.signUp() спрацьовує SQL-тригер, який автоматично створює запис у таблиці профілів.

**2. Автентифікація (Authentication)**

- **Процес входу:** Реалізований через метод signInWithPassword(). При успішному вході сервер повертає JWT-токен (JSON Web Token).
- **Управління сесією:** Клієнтська бібліотека @supabase/supabase-js автоматично зберігає токен у localStorage та додає його в заголовок Authorization: Bearer <token> для кожного запиту до бази даних.
- **Глобальний стан:** У додатку реалізовано AuthContext.jsx, який за допомогою хука onAuthStateChange відстежує стан сесії (логін/вихід) у реальному часі та надає дані про користувача всім компонентам.

**3. Авторизація (Authorization) та Middleware**

Авторизація реалізована на двох рівнях для забезпечення концепції "Defense in Depth" (глибинний захист):

- **Frontend (React Auth Guards):** Створено компонент-обгортку ProtectedRoute.jsx. Він перевіряє наявність активної сесії у контексті. Якщо користувач не авторизований, компонент блокує рендеринг сторінки та перенаправляє на /login.
- **Backend (PostgreSQL Row Level Security):** Доступ до таблиці tasks обмежено на рівні СУБД. Навіть якщо зловмисник отримає доступ до API, політика (auth.uid() = user\_id) фізично не дозволить отримати дані, що належать іншому UUID.

**4. RBAC (Рольова модель) та захист Адмін-панелі**

- **Ролі:** У таблиці profiles визначено ролі user та admin.
- **Додатковий захист (Lock Screen):** Маршрут /admin захищений подвійно:
  - Перевірка ролі користувача в метаданих сесії.
  - **Master Password:** Перед відображенням списку користувачів реалізовано екран блокування, який вимагає введення адміністративного пароля. Це запобігає несанкціонованому доступу до панелі управління, якщо сесія адміна залишилася відкритою.

**5. Перевірка сценаріїв безпеки**

- **Ізоляція даних:** Спроба доступу до чужої задачі (шляхом підміни ID в URL або запиті) ігнорується базою даних завдяки RLS.
- **Блокування доступу:** В адмін-панелі реалізовано функціонал перемикання прапорця is\_blocked. Система перевіряє цей статус під час входу.
- **Захист профілю:** Сторінка /profile дозволяє користувачу редагувати лише власне поле full\_name, при цьому поля email та role доступні лише для читання.

**Висновок**

У межах ЛР №4 реалізовано надійний цикл безпеки. Використання Supabase Auth забезпечило безпечну роботу з сесіями через JWT, а комбінація клієнтських "гардів" (ProtectedRoute) та серверних політик (RLS) гарантувала сувору авторизацію. Додатковий рівень захисту адмін-панелі паролем та рольова модель доступу роблять систему стійкою до поширених вразливостей вебдодатків.
