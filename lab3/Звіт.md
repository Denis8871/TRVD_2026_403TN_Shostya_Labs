**Мета роботи:**

Реалізувати взаємодію з даними (Data Access Layer), бізнес-логіку (Service Layer) та інтеграцію з REST API на базі хмарної інфраструктури Supabase (BaaS) для додатку Task Manager.

**1. Реалізація Data Access Layer (DAL)**

Оскільки використовується архітектура Backend-as-a-Service, роль класичної ORM виконує клієнт @supabase/supabase-js, який генерує запити до PostgreSQL.

- **Створені таблиці у базі даних Supabase:**
  - profiles (зв'язана з auth.users через SQL-тригер)
  - tasks
- **CRUD-операції інкапсульовані в окремих модулях на стороні клієнта:**
  - src/api/supabase.js — ініціалізація єдиного клієнта бази даних (Singleton).
  - src/api/taskService.js — абстракція для роботи з таблицею задач (методи getTasks, createTask, updateTask, deleteTask).

**2. Реалізація Service Layer**

Бізнес-логіка та правила доступу реалізовані на двох рівнях:

- **На рівні Бази Даних (Row Level Security - RLS):**

  Налаштовано політики (Policies) у Supabase, які гарантують, що користувач може читати, створювати, оновлювати та видаляти лише свої власні задачі (auth.uid() = user\_id). Це замінює класичні перевірки прав у контролерах бекенду.

- **На рівні Клієнтського додатку (React Services / Context):**
  - src/api/taskService.js — обробляє дані перед відправкою до БД, формує запити та обробляє помилки від API.
  - src/context/AuthContext.jsx — інкапсулює логіку управління сесією користувача, логіном/реєстрацією (через supabase.auth) та очищенням локального стану при виході.

**3. Реалізація API (PostgREST)**

Supabase автоматично генерує RESTful API для створених таблиць за допомогою PostgREST.

**Підтримані ендпоінти для задач:**

Усі запити виконуються до базового URL проєкту Supabase з передачею JWT-токена в заголовку Authorization.

- GET /rest/v1/tasks?select=\* — отримання списку задач поточного користувача.
- POST /rest/v1/tasks — створення нової задачі.
- PATCH /rest/v1/tasks?id=eq.{task\_id} — оновлення статусу, пріоритету або тексту задачі.
- DELETE /rest/v1/tasks?id=eq.{task\_id} — видалення задачі.

**Статус-коди відповідей:**

- 200 OK — успішне читання/оновлення.
- 201 Created — успішне створення задачі.
- 204 No Content — успішне видалення.
- 400 Bad Request — помилка валідації (наприклад, невірний формат UUID).
- 401 Unauthorized — запит без JWT-токена або час його дії минув.
- 403 Forbidden — RLS політика заборонила доступ (спроба доступу до чужої задачі).

**4. DTO та мапінг**

У JavaScript/React екосистемі відсутні жорсткі класи DTO, але принцип трансформації даних збережено:

- **Вхідні дані (Request Mappers):** Перед відправкою до Supabase дані з форм доповнюються необхідними полями. Наприклад, у компоненті Dashboard.jsx під час створення задачі до об'єкта примусово додається ID поточного користувача: await taskService.createTask({ ...taskData, user\_id: user.id });.
- Трансформація даних відбувається безпосередньо у методах taskService.js перед викликом API.

**6. Як запустити і перевірити**

Оскільки вся серверна частина знаходиться у хмарі (Supabase), локальний запуск зводиться до старту клієнтського додатку:

1. Клонувати репозиторій.
1. Створити файл .env.local у корені проєкту та додати туди ключі VITE\_SUPABASE\_URL і VITE\_SUPABASE\_ANON\_KEY.
1. Виконати npm install.
1. Виконати npm run dev.
1. Відкрити http://localhost:5173 у браузері.
