
**1. REST, GraphQL, gRPC — коли що використовувати?**

**REST**

- стандартний підхід на базі HTTP
- ресурси (users, orders), CRUD-операції

**Коли використовувати:**

- прості та середні API
- публічні API
- коли важлива простота і зрозумілість

**GraphQL**

- клієнт сам визначає, які дані отримати
- один endpoint замість багатьох

**Коли використовувати:**

- складні фронтенди (наприклад, SPA)
- коли потрібно гнучко отримувати дані
- багато різних клієнтів (web, mobile)

**gRPC**

- використовує Protocol Buffers
- швидкий і ефективний (бінарний формат)
- підтримує стрімінг

**Коли використовувати:**

- мікросервіси
- high-load системи
- внутрішня комунікація між сервісами

**2. Що таке over-fetching та under-fetching?**

**Over-fetching**

- отримуєш більше даних, ніж потрібно\
  Приклад: REST endpoint повертає весь профіль, хоча потрібне лише ім’я

**Under-fetching**

- отримуєш недостатньо даних і робиш кілька запитів\
  Приклад: окремо отримуєш користувача, потім його замовлення

**GraphQL вирішує обидві проблеми**, бо дозволяє запитувати тільки потрібні поля.

**3. Що таке Protocol Buffers і чим краще за JSON?**

**Protocol Buffers (Protobuf)** — це бінарний формат серіалізації даних від Google.

**Переваги над JSON:**

- менший розмір (економія трафіку)
- швидша серіалізація/десеріалізація
- чітка схема (.proto файл)
- типобезпечність

**Недоліки:**

- нечитабельний для людини
- складніше дебажити

**4. Як версіонують API у REST і GraphQL?**

**REST:**

- у URL: /api/v1/users
- у заголовках (Header versioning)
- через параметри

Найпоширеніше — версія в URL.

**GraphQL:**

- зазвичай **без версій**
- використовують:
  - deprecation (застарілі поля)
  - додавання нових полів без лому старих

Ідея: еволюція схеми без створення нових версій.

**5. Що таке idempotency і які методи мають бути ідемпотентними?**

**Idempotency** — це властивість, коли повторний виклик дає той самий результат, що і перший.

**Ідемпотентні HTTP-методи:**

- GET — тільки читання
- PUT — повне оновлення
- DELETE — видалення (повторний виклик нічого не змінює)

**Не ідемпотентний:**

- POST — створює нові ресурси (кожен виклик може створити новий об’єкт)

**6. Яка роль SDL або .proto файлів?**

**SDL (Schema Definition Language) — GraphQL:**

- описує типи, запити, мутації
- є контрактом між клієнтом і сервером

**.proto файли — gRPC / Protobuf:**

- описують структуру даних і сервіси
- генерують код для клієнта і сервера

Коротко:

- SDL і .proto — це **контракти API**
- вони визначають, як сервіси взаємодіють між собою
